# SPEC.md — 機能仕様

## 1. プロジェクト概要

### 1.1 目的

スマホ・PCから作成したHTMLファイル（単一HTML、または関連ファイル込みのzip）を、ブラウザから直接GitHubリポジトリにアップロードし、URLを知っている人なら誰でも閲覧できる個人用の静的サイトを構築する。

### 1.2 設計原則

- **サーバーレスで完結**: バックエンドサーバーを持たず、GitHub Pages + GitHub APIで実現
- **モバイルファースト**: 平日はスマホで気軽にアップロード、週末にPCで整理する運用を想定
- **完全無料**: Public repoの範囲内で全機能を実現
- **データの透明性**: メタデータはmanifest.jsonとして人間が読める形式で保持

### 1.3 ユーザーストーリー

| # | ストーリー |
|---|---|
| US-01 | 運用者として、外出先のスマホから作成したHTMLをすぐにアップロードして、後で見返せるようにしたい |
| US-02 | 運用者として、HTML + 画像のような複数ファイルをzipでアップロードしたい |
| US-03 | 運用者として、アップロードしたファイルをフォルダで整理し、タグで分類したい |
| US-04 | 運用者として、検索や並べ替えで目的のファイルをすぐ見つけたい |
| US-05 | 運用者として、不要になったファイルを削除できるようにしたい |
| US-06 | 閲覧者として、URLを共有されたファイルをブラウザで普通に閲覧したい |
| US-07 | 運用者として、サイト全体がGoogleにインデックスされないようにしたい |
| US-08 | 運用者として、将来別の運用者を追加できるようにしたい |

### 2.2 機能詳細

#### F-01: 単一HTMLアップロード

- ユーザーはファイル選択UIから `.html` ファイルを1つ選ぶ
- 入力フィールド:
  - 表示名（必須、デフォルトはファイル名）
  - フォルダ（既存フォルダから選択 or 新規入力）
  - タグ（複数、カンマ区切り、既存タグからサジェスト）
- 「アップロード」ボタン押下で:
  1. ULIDを発行（時系列ソート可能なID）
  2. HTMLの `<head>` に `<meta name="robots" content="noindex, nofollow">` を自動挿入（既存があればスキップ）
  3. 物理パス: `/files/{ulid}/index.html` として配置
  4. manifest.jsonに新エントリを追加
  5. ファイル + manifest更新を**単一commit**としてGit Data APIでpush
  6. UI上でcommit成功 → Pagesデプロイ待ち（30秒〜2分）の表示
  7. 完了後、閲覧用URLを表示

#### F-02: zipアップロード（自動展開）

- ユーザーは `.zip` ファイルを1つ選ぶ
- クライアントサイドでJSZipを使って展開
- エントリポイント検出:
  1. ルートに `index.html` があればそれを採用
  2. ルートになければ階層下を探索、唯一の `.html` ファイルがあれば自動採用
  3. 候補が複数ある場合、UI上で選択させる
- 入力フィールド:
  - 表示名（必須、デフォルトはzipファイル名）
  - フォルダ・タグ（F-01と同じ）
- 「アップロード」ボタン押下で:
  1. ULIDを発行
  2. エントリポイントHTMLに noindex meta を自動挿入
  3. 物理パス: `/files/{ulid}/` 配下にzip内ファイル構造をそのまま展開配置
  4. エントリポイントのパスを manifest.json の `entryPath` フィールドに記録（ルートでなければ `subfolder/index.html` 等）
  5. 全ファイル + manifest更新を**単一commit**としてGit Data APIでpush
  6. 以降はF-01と同様

#### F-03: 一覧表示

- レイアウト: グリッドのサムネカード
  - PC: 3〜4列
  - タブレット: 2〜3列
  - スマホ: 1〜2列
- カード要素:
  - サムネ画像（16:9、未生成時はプレースホルダ）
  - 表示名
  - フォルダパス（パンくず形式）
  - タグ（チップ表示、最大3つ、超過時は `+N`）
  - 更新日時（相対表記、ホバーで絶対表記）
- カードクリック: 新タブで `/files/{ulid}/{entryPath}` を開く

#### F-04: フォルダナビゲーション

- 左サイドバー（PC） / ハンバーガーメニュー（スマホ）でフォルダツリー
- ツリーの構造はmanifest.jsonから動的に集約
- フォルダクリックで一覧をそのフォルダ配下のファイルに絞り込み
- ブレッドクラム表示で現在位置を明示
- ルートには「すべて」ノードがある

#### F-05: タグフィルタ

- 上部にタグクラウド（出現頻度順）
- タグクリックで該当タグを持つファイルに絞り込み
- 複数タグ選択時はAND条件
- 選択解除は再クリック

#### F-06: キーワード検索

- 上部に検索ボックス
- 表示名、フォルダパス、タグを対象に部分一致検索
- 大小文字無視
- 全角・半角は区別しない（normalize）
- インクリメンタル検索（入力ごとに即時フィルタ）

#### F-07: 並べ替え

- ソートキー: 更新日時 / アップロード日時 / 表示名 / サイズ
- 昇順・降順切り替え
- デフォルト: 更新日時 降順

#### F-08: リネーム

- 一覧で対象のカードに「⋮」メニュー → 「リネーム」
- インライン編集 or モーダル
- 表示名のみ変更（物理ファイル名は不変）
- 確定 → manifest.json更新のcommit

#### F-09: フォルダ移動

- 「⋮」メニュー → 「フォルダ移動」
- フォルダパス入力（既存サジェスト + 新規可）
- 確定 → manifest.json更新のcommit

#### F-10: タグ編集

- 「⋮」メニュー → 「タグ編集」
- タグの追加・削除（チップUI）
- 確定 → manifest.json更新のcommit

#### F-11: 削除

- 「⋮」メニュー → 「削除」
- 確認モーダル（「この操作は取り消せません」表示）
- 確定 → `/files/{ulid}/` 配下全削除 + manifest.json該当エントリ削除を**単一commit**でpush
- サムネ画像（`/thumbnails/{ulid}.png`）も削除

#### F-12: PAT管理

- `/manage/setup` 画面
- PAT入力フィールド（パスワード型表示）
- 「疎通テスト」ボタン → リポジトリ情報取得APIで検証
- 保存 → localStorageに `htmlvault.pat` キーで保存
- 既存PATがある場合は「更新」「削除」ボタン表示
- 削除時はlocalStorageから消去

#### F-13: サムネ自動生成

- 詳細は `ARCHITECTURE.md` のActions設計を参照
- 動作概要:
  - `files/**` への変更をトリガー
  - 各ファイルについて、HTMLの `<head>` から `og:image` を抽出 → あれば取得して使用
  - なければPuppeteerでスクショ撮影（1280x720）
  - `/thumbnails/{ulid}.png` として保存
  - manifest.jsonの該当エントリの `thumbnail` フィールド更新
  - 自動commit（メッセージに `[skip ci]` を含めて再トリガー回避）

#### F-14: noindex meta自動付与

- アップロード処理（F-01、F-02）の一環として、HTML本体に以下を自動注入:
  ```html
  <meta name="robots" content="noindex, nofollow">
  ```
- `<head>` タグの先頭に挿入。既に同タグがあればスキップ
- `<head>` がない場合は作成して挿入
- リポジトリルートには `/robots.txt` を配置:
  ```
  User-agent: *
  Disallow: /
  ```

#### F-15: 複数選択・一括操作

- カードにチェックボックス（hover時 or 選択モード時表示）
- 「一括操作」モード切り替えボタン
- 一括操作: 削除 / フォルダ移動 / タグ追加・削除
- 一括操作は**単一commit**でまとめて実行

#### F-16: キーボードショートカット

| キー | 動作 |
|---|---|
| `/` | 検索ボックスにフォーカス |
| `n` | 新規アップロード画面を開く |
| `Esc` | モーダル/メニューを閉じる |
| `Ctrl/Cmd + A` | 一覧表示中、全選択（選択モード時のみ） |
| `Delete` | 選択中ファイルを削除（確認モーダル経由） |

#### F-17: ドラッグ&ドロップアップロード

- `/manage` 画面で、ファイルを画面にドロップするとアップロードフローに進む
- 複数ファイルドロップ時は1つずつ順次処理
- 視覚フィードバック（ドロップエリアハイライト）

#### F-18: PWA対応

- vite-plugin-pwaでService Worker生成
- manifest.json（PWA用、サイトのmanifestとは別物）にアイコン・名前を設定
- ホーム画面に追加可能
- iOS Safariでホーム画面起動した場合、localStorageは安定して保持される

#### F-19: アップロード進行状況表示

- アップロード中:
  - 「ファイル読み込み中」
  - 「noindex注入中」
  - 「コミット作成中」
  - 「Push中」
- 完了後:
  - 「コミット完了。Pagesデプロイ待ち（通常30秒〜2分）」
- プログレスバー or スピナーで視覚化

#### F-20: Pagesデプロイ反映待ち表示

- commit成功後、Pagesデプロイ状態を **GitHub Actions API** でポーリング
  - エンドポイント: `/repos/{owner}/{repo}/pages/builds/latest`
  - 状態: `building` → `built` で完了判定
- 完了後、閲覧用URL（クリック可能）を表示

## 3. データモデル

### 3.1 manifest.json

リポジトリのルートに配置。全ファイルのメタデータを集約。

```json
{
  "version": 1,
  "updatedAt": "2026-06-20T12:34:56.000Z",
  "files": [
    {
      "id": "01HQXYZ1234567890ABCDEFGHJ",
      "displayName": "雀の舞 設計メモ",
      "folder": "/projects/sparrow",
      "tags": ["design", "novel", "wip"],
      "entryPath": "index.html",
      "kind": "single",
      "uploadedAt": "2026-06-20T12:34:56.000Z",
      "updatedAt": "2026-06-20T12:34:56.000Z",
      "size": 12345,
      "uploader": "soya-github-username",
      "thumbnail": "/thumbnails/01HQXYZ1234567890ABCDEFGHJ.png"
    }
  ]
}
```

### 3.2 フィールド定義

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `version` | number | ✅ | スキーマバージョン。マイグレーション用 |
| `updatedAt` | ISO8601 string | ✅ | manifest最終更新日時 |
| `files[]` | array | ✅ | ファイルエントリの配列 |
| `files[].id` | string | ✅ | ULID（26文字）。物理ディレクトリ名としても使用 |
| `files[].displayName` | string | ✅ | 表示名。一覧UIで表示。リネーム対象 |
| `files[].folder` | string | ✅ | 仮想フォルダパス。ルートは `""`（空文字列） |
| `files[].tags` | string[] | ✅ | タグの配列。空配列可 |
| `files[].entryPath` | string | ✅ | `/files/{id}/` からの相対パス。単一HTMLは `"index.html"` |
| `files[].kind` | `"single" \| "zip"` | ✅ | アップロード元の種別 |
| `files[].uploadedAt` | ISO8601 string | ✅ | 初回アップロード日時 |
| `files[].updatedAt` | ISO8601 string | ✅ | メタデータ最終更新日時 |
| `files[].size` | number | ✅ | エントリポイントHTMLのバイトサイズ |
| `files[].uploader` | string | ✅ | アップロードしたGitHubユーザー名 |
| `files[].thumbnail` | string \| null | ✅ | サムネ画像パス。未生成時はnull |

### 3.3 ID生成（ULID）

- 26文字のCrockford's Base32
- 先頭10文字がタイムスタンプ（48bit ms精度）→ 時系列ソート可能
- 残り16文字がランダム
- 衝突確率は実質ゼロ
- ライブラリ: `ulid` npm package

### 3.4 フォルダパスの表記規則

- Unix風スラッシュ区切り: `/projects/sparrow`
- ルートは空文字列 `""` または `"/"`（内部的に正規化）
- 末尾スラッシュなし（正規化）
- 大文字小文字は区別する

## 5. 非機能要件

| 項目 | 要件 |
|---|---|
| パフォーマンス | 初回ロード3秒以内（3G想定）、操作レスポンス 100ms以内 |
| 対応ブラウザ | Chrome / Safari / Edge / Firefox 最新版 |
| 対応OS | iOS 16+ / Android 12+ / Windows 10+ / macOS 12+ |
| アクセシビリティ | WAI-ARIA基本準拠（フォーカス可、スクリーンリーダー対応） |
| 国際化 | 日本語のみ（i18n対応は将来検討） |
| データ容量 | 1ファイル最大100MB（GitHubの制限）、推奨50MB以下 |
| ファイル数 | 数千件まで実用的に動作 |

## 6. 制約事項

- GitHub API レート制限: 認証済み 5,000 req/時。通常使用では問題なし
- GitHub Pages 帯域: 100GB/月（無料枠）。個人利用なら十分
- 1ファイル最大100MB（GitHub git制限）
- リポジトリ全体推奨5GB以下
- Pagesデプロイ反映時間: commit後30秒〜2分

## 7. セキュリティ要件

- PATは localStorage に保存（端末ごと、`htmlvault.pat` キー）
- PATは送信時以外マスク表示
- HTTPS必須（GitHub Pagesは標準でHTTPS）
- 外部スクリプトを読み込まない（XSS対策）
- React の `dangerouslySetInnerHTML` 禁止
- アップロードされるHTMLは別オリジン扱い（CSPで隔離はせず、noindex対応のみ）
